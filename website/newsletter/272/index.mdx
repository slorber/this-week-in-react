**[Next.js Source Maps: From Random Chunks to Real Code](https://blog.sentry.io/setting-up-next-js-source-maps-sentry/?utm_source=thisweekinreact&utm_medium=paid-community&utm_campaign=nextjs-fy27q1-nextjs&utm_content=newsletter-first-sponsor-blog-sourcemaps-learnmore)**

When [Next.js builds your app](https://sentry.io/for/nextjs/?utm_source=thisweekinreact&utm_medium=paid-community&utm_campaign=nextjs-fy27q1-nextjs&utm_content=newsletter-first-sponsor-product-trysentry), it compiles and minifies your code into chunks like **static/chunks/12345-something.js**. When errors happen, your stack traces point there. Not to the code you actually wrote.

This blog walks through how **source maps** and **debug IDs** wire things together, and how to configure Sentry to show real file names and line numbers in production.

*   What actually happens to your code during build
*   Why dev tools show readable traces but Sentry shows chunks
*   How to simulate a production build so Sentry gets your source maps
*   Troubleshooting missing or stale source maps
